# Playbook — Ингест материалов в KB

> Этот документ — самодостаточная инструкция для отдельной сессии ингеста. Свежий чат, открытый для наполнения базы, читает его и следует по шагам, не теряя архитектурных принципов.

## 0. Перед началом (обязательное чтение)

Прежде чем что-либо извлекать, прочитай в таком порядке:

1. `../../../00_Архитектура_базы_знаний.md` — концепция (особенно слои 3-5, сквозные механизмы, тиерная валидация). Лежит на уровне проекта, выше репо.
2. `../constitution/Constitution.md` — **методологическая линза Pavel'я**. Без неё нельзя решать, что подходит методу, а что «вне рамки». Если Constitution ещё не написан — ингест внешних книг НЕ начинать (можно только собственные материалы Pavel'я как сырьё).
3. Шаблоны в `../00 Мета/Шаблоны/` — точная схема карточек.
4. Текущую онтологию в `../ontology/` — какие школы и парадигмы уже известны.

## Назначение

Превратить материал в атомарные карточки с соблюдением четырёх правил:

- **Anti-echo:** карточки извлекаются ТОЛЬКО из сырого источника, не из существующих карточек. Существующие карточки используются исключительно для проверки на дубликат.
- **Provenance:** каждая карточка несёт `extracted_by`, ссылку на источник, `confidence`, `last_verified`.
- **Тиерная валидация:** AI создаёт только `draft`-карточки. Повышение статуса, новые узлы онтологии, конфликты, мерджи, удаления — в `kb-ops/inbox/` на review Pavel'я.
- **Парадигмальный страж:** при обнаружении противоречия между парадигмами — НЕ синтезировать, создавать `concept-conflict` карточку.

## Триггер

Вручную из Cowork-сессии. Пример запроса: «Ингести материал `sources/raw/моя-статья-о-границах.md` как собственный материал».

## Корпусная стратегия (порядок важен)

Если ингестятся несколько материалов в рамках первичного наполнения, соблюдай порядок из концепции:

1. **Сначала собственные материалы Pavel'я** — они образуют canonical-ядро (после его review). Каждая карточка из них получает приоритет в retrieval.
2. **Потом внешние книги** — проходят двойной фильтр: extract → сверка с canonical-ядром (дубликат / расширение / конфликт) → review.

Тип материала (`собственный` / `внешний`) определяет дефолтный трек обработки на шаге 2.

## Шаги

### Шаг 1. Подготовка материала

1. Определи формат. Если не `.md`:
   - PDF/EPUB → конвертация в markdown (инструмент выбрать при первом столкновении: pandoc для EPUB, marker/marker-pdf для PDF со сложной вёрсткой). Конвертированный .md кладётся в `sources/raw/`.
   - Аудио/видео → расшифровка (Whisper или аналог), результат в `sources/raw/`.
2. Для **книг** — предобработка чанкинга: убедись, что текст структурирован заголовками H1–H3 по смысловым секциям. MCP-чанкер режет по H1–H3 без overlap (max 2000 символов), поэтому границы должны попадать по структуре источника. При необходимости — расставить заголовки вручную.
3. Только `.md` индексируется MCP. Бинарники (оригинальные PDF) хранить рядом, но не рассчитывать на их индексацию.

### Шаг 2. Регистрация источника

1. Создай source-запись в `sources/` по шаблону `source.md`:
   - `id` = slug (совпадает с именем файла)
   - `authors`, `year`, `language` (ru/en), `school`, `paradigm`
   - `source_type` (book | article | lecture | transcript | own_material | course)
   - `status_kind` (primary | retelling | criticism)
   - `themes` — ключевые темы
2. Если есть полный текст — положи его как `sources/raw/{id}-full.md` (он проиндексируется, но карточки строятся НЕ из него повторно — anti-echo).
3. **Собственный материал Pavel'я** → source помечается `source_type: own_material`, `status_kind: primary`. Карточки из него — кандидаты в canonical-ядро (но всё равно создаются как `draft`, канонизирует Pavel).

### Шаг 3. Извлечение карточек (anti-echo)

1. Читай **сырой источник напрямую** (текст из шага 1), НЕ существующие карточки.
2. Выдели кандидаты: концепции, техники, фреймворки, модели. Для каждого — определи тип (concept | technique | framework).
3. Для каждого кандидата ПЕРЕД созданием — проверка на дубликат (шаг 4).
4. Если кандидат новый — создай `draft`-карточку по соответствующему шаблону:
   - `status: draft`, `confidence: low`
   - `extracted_by: "claude-<model> <дата>"`
   - `sources: ["[[{id источника}]]"]`
   - `school` — чистое имя без `[[]]` (для фильтруемости); wikilink — в теле
   - `paradigm` — одно значение из enum
   - применяй **линзу Constitution**: если концепция методически чужда рамке Pavel'я — всё равно создай карточку, но в теле явно пометь «вне рамки метода» и НЕ ставь её в `related` к canonical-карточкам Pavel'я.

### Шаг 4. Проверка на дубликат (dedup)

Для каждого кандидата перед созданием:

1. `semantic_search` по kb-knowledge с запросом = суть кандидата.
2. `search_by_tag` / проверка `aliases` существующих карточек.
3. Решение:
   - **Точного дубликата нет** → создаём draft (шаг 3.4).
   - **Есть очень похожая карточка** → НЕ создаём дубль, НЕ правим существующую (особенно canonical). Создаём запись в `kb-ops/inbox/` с предложением «расширить карточку X данными из источника Y» — на review Pavel'я.

### Шаг 5. Проверка онтологии

Для каждой новой карточки:

1. Существует ли узел школы (`ontology/schools/{school}.md`) и парадигмы (`ontology/paradigms/{paradigm}.md`)?
2. Если узла НЕТ → **эскалация (human-checkpoint)**: создать запись в `kb-ops/inbox/` «новая школа/парадигма {X}, обнаружена при ингесте {источник}». НЕ создавать узел онтологии автоматически. Карточку оставить с `status: draft`, пометить в inbox-записи зависимость.

### Шаг 6. Детекция конфликтов (парадигмальный страж)

Для каждой новой карточки:

1. `semantic_search` среди `canonical`-карточек (`where_metadata={"fm_status":"canonical"}`) по сути новой карточки.
2. Если найдена карточка, которая **методически противоречит** новой (разные действия коуча в одной ситуации из-за разной трактовки):
   - НЕ синтезировать «общий взгляд».
   - Создать `concept-conflict` draft-карточку по шаблону `card-conflict.md` (взгляд A / взгляд B / где конфликтует / позиция метода — пусто, на Pavel'я).
   - Запись в `kb-ops/inbox/`.

### Шаг 7. Валидация схемы

```bash
cd "<vaults>" && python3 scripts/validate_frontmatter.py kb-knowledge
```

- Ошибки схемы (missing fields, неверные enum) — AI чинит сам (это auto-tier: форматирование/заполнение полей).
- Должно быть 0 errors перед индексацией.

### Шаг 8. Индексация и проверка

1. `reindex(vault="kb-knowledge", force=false)` (или `force=true` если правил много).
   - Помни операционную находку 0.2.2: свежедобавленные vault'ы могут требовать `force=true`.
2. Sanity-check: `semantic_search` с `where_metadata` по `fm_type`/`fm_status` — убедись, что новые карточки находятся и фильтруются.

### Шаг 9. Audit + inbox

1. Запись в `kb-ops/audit/YYYY-MM-DD.md` по шаблону `audit-entry.md`:
   - `action: ingest`, `operator`, `target_vault: kb-knowledge`
   - `target_paths` — список созданных карточек
   - `reasoning` — из какого источника, сколько карточек, сколько эскалаций
2. Все эскалации шагов 4-6 — отдельными файлами в `kb-ops/inbox/`.

### Шаг 10. Закрытие батча

Отчёт Pavel'ю:

- N draft-карточек создано (по типам)
- M потенциальных мерджей в inbox
- K конфликтов помечено
- L пробелов онтологии эскалировано
- Что в `inbox/` ждёт его review
- Напоминание: коммит (`git add` + `git commit`) и `git push` — действие Pavel'я (sandbox без credentials) ЛИБО выполняется в той сессии, если credentials доступны.

## Что остаётся AUTO, что идёт в REVIEW

| Auto (AI делает сам) | Review (inbox → Pavel) |
|---|---|
| Создание draft-карточек | Промоция `draft → reviewed → canonical` |
| Source-записи | Изменение существующей `canonical`-карточки |
| Форматирование, заполнение полей | Мердж двух карточек |
| Дедуп точных дубликатов | Новый узел онтологии (школа/парадигма) |
| Валидация и починка схемы | Карточки `concept-conflict` |
| Переиндексация | Удаление чего-либо |
| Audit-записи | Понижение/повышение confidence вручную |

## Эскалация (human-checkpoint triggers)

AI обязан остановиться и создать `inbox/`-запись, не действуя сам, при:

1. Новой школе / авторе / парадигме вне онтологии
2. Противоречии с canonical-карточкой
3. Совпадении с существующей карточкой (потенциальный merge)
4. Низком confidence на материале для client-facing задач

## Acceptance (когда плейбук отработал корректно)

- Все созданные карточки прошли `validate_frontmatter.py` (0 errors)
- Каждая карточка имеет provenance (`extracted_by`, `sources`, `confidence`, `status: draft`)
- Ни одна `canonical`-карточка не изменена автоматически
- Ни один узел онтологии не создан автоматически
- Все четыре типа эскалаций, если возникли, попали в `kb-ops/inbox/`
- Есть audit-запись за день
- Новые карточки находятся через `semantic_search` + `where_metadata`

## Известные ограничения / на доработку

- Конвертеры PDF/EPUB/аудио пока не зафиксированы — выбрать при первом реальном материале, потом дописать сюда.
- Формат audit (один файл на день vs папка на день) — выбрать на первом батче, зафиксировать.
- Порог confidence для client-facing — пока качественный (`canonical`/`reviewed`), численный не введён.
